# Centralize Entity Lifecycle Timestamp Ownership

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. Maintain this document in accordance with `.agent/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, every Book or Author mutation will have one clear owner for entity lifecycle time: the use-case interactor coordinating the operation. New entities created by one operation will receive identical `created_at` and `updated_at` values, updates will preserve `created_at`, and history restoration will retain the historical creation time while marking the entity updated at restore time. GraphQL clients will see the same fields and formats as before; tests demonstrate the corrected semantics.

## Surprises & Discoveries

- Observation: Imported Books already share one interactor-generated timestamp, but Authors created by the same import use PostgreSQL column defaults.
  Evidence: `ImportBooksInteractor::import` builds every `ImportBookInput` with one local `now`, while `PgAuthorRepository::find_or_create_by_name` omits `created_at` and `updated_at` from its insert.

- Observation: Event timestamps already have a distinct audit meaning and are generated by PostgreSQL.
  Evidence: the event migration defines `event_set.created_at` and entity-event `changed_at` with `DEFAULT current_timestamp`.

## Decision Log

- Decision: Do not introduce a Clock trait.
  Rationale: Explicit timestamp parameters make domain logic deterministic, while interactors only need one real UTC reading per operation. The additional dependency-injection surface is not justified by current requirements.
  Date/Author: 2026-07-22 / Codex

- Decision: Keep database audit timestamps separate from entity lifecycle timestamps.
  Rationale: Audit timestamps describe when PostgreSQL recorded an event; lifecycle timestamps describe when the use case changed entity state. A small difference is meaningful and acceptable.
  Date/Author: 2026-07-22 / Codex

- Decision: Restore historical `created_at` and replace historical `updated_at` with the restore operation time.
  Rationale: Restore is a new mutation of the current entity while its original creation identity remains intact.
  Date/Author: 2026-07-22 / User and Codex

## Context and Orientation

The repository is a Rust API using domain entities, use-case interactors, repository traits, and PostgreSQL repository implementations. `src/domain/entity/author.rs` defines Author constructors and currently reads the system clock inside `Author::new_with_yomi`. `src/use_case/interactor/book.rs` and `src/use_case/interactor/author.rs` coordinate mutations and already read current time for most paths. `src/use_case/interactor/event.rs` restores historical snapshots. `src/domain/repository/author_repository.rs` defines author persistence, and `src/infrastructure/author_repository.rs` implements it with SQLx.

An entity lifecycle timestamp is `created_at` or `updated_at` stored on a Book or Author. An audit timestamp is `event_set.created_at` or an entity event's `changed_at`; it records database event creation and is intentionally outside this change. `normalize_timestamp_for_persistence` truncates entity timestamps to PostgreSQL's microsecond precision so an in-memory value equals its stored round trip.

The transaction boundary remains owned by interactors through `TransactionManager`. This change does not alter transactions, GraphQL schema, database schema, event creation, or timestamp serialization.

For a non-delete history event, read one restore time, build the entity with the event's historical `created_at`, and supply the restore time as `updated_at`. The PostgreSQL insert explicitly writes both lifecycle columns and returns them. The conflict branch must remain non-mutating so existing Author timestamps are unchanged. ## Validation and Acceptance

Fixed domain tests must show that Author creation with a supplied timestamp produces equal normalized lifecycle fields. Mutation interactor tests must show creation timestamps lie within the operation interval and are equal, and updates retain their old creation time while receiving a new update time. Restore tests must use a historical creation and update pair and verify the returned and persisted entity retains historical creation but does not retain historical update time. Import repository coverage must show an inserted Author has both supplied timestamps while an existing Author's timestamps remain unchanged.

The change is accepted when the mandatory formatting, Clippy, and full locked test commands all pass; `git diff` contains no GraphQL schema or migration changes; all OpenSpec tasks are checked; and a Draft PR against `main` exists for the pushed feature branch.

## Interfaces and Dependencies

No new crate dependency is added. `time::OffsetDateTime` remains the timestamp type.

At completion, `Author::new` and `Author::new_with_yomi` accept a creation timestamp used for both lifecycle fields. `Author::new_with_timestamps` continues to accept separate persisted creation and update timestamps. `AuthorRepository::find_or_create_by_name` accepts `created_at: OffsetDateTime` after the author name argument. Its insert behavior uses that value for both `created_at` and `updated_at`; its existing-row behavior does not update either field.
