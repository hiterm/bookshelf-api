## Context

Book and Author lifecycle timestamps have the same persistence shape but different ownership. Author convenience constructors call `OffsetDateTime::now_utc()` inside the domain, Book interactors supply timestamps, and `find_or_create_by_name` lets PostgreSQL defaults timestamp authors created during import. Update interactors already pass an application-generated timestamp into the entities. Event audit timestamps (`event_set.created_at` and entity-event `changed_at`) are separate values generated by PostgreSQL when rows are recorded.

## Goals / Non-Goals

**Goals:**

- Give the use-case layer sole responsibility for choosing entity lifecycle timestamps.
- Use one timestamp for every entity created by one logical operation.
- Give restore operations explicit lifecycle semantics.
- Preserve persisted timestamps exactly on repository reads.

**Non-Goals:**

- Introducing a Clock trait or dependency.
- Making entity lifecycle timestamps equal database audit timestamps.
- Changing GraphQL timestamp fields, serialization, database columns, or precision.

## Decisions

- Each mutating interactor obtains `OffsetDateTime::now_utc()` once after input validation and before constructing the affected entity state. Passing time explicitly keeps domain behavior deterministic without adding generic Clock dependencies throughout dependency injection. A Clock abstraction can be introduced later if exact interactor-time tests or alternative time sources become requirements.
- New Author constructors accept one creation time and assign it to both lifecycle fields. The persistence reconstruction constructor continues to accept both values. This prevents normal creation from accidentally producing distinct creation and update times while retaining exact database reconstruction.
- Existing Book construction remains capable of accepting two timestamps because it also serves persistence and history reconstruction. Normal Book creation supplies the same local `now` for both fields.
- Restore keeps the historical `created_at` but replaces historical `updated_at` with the restore operation time. A restore is a new lifecycle mutation, while event history still contains the prior snapshot and the database-generated audit time for the restore event.
- `AuthorRepository::find_or_create_by_name` accepts the import operation time. Its insert path explicitly stores that time in both columns; its conflict path returns the existing row without changing timestamps. Moving lookup-or-create into the use case was rejected because atomic conflict handling belongs in persistence.
- Timestamp normalization stays at the domain boundary through `normalize_timestamp_for_persistence`. Although the microsecond precision reflects PostgreSQL storage, retaining it prevents in-memory mutation results from differing from immediate persisted round trips.
- PostgreSQL continues to generate `event_set.created_at` and entity-event `changed_at`. These represent when audit records were written, not the entity's effective lifecycle time, so small differences are intentional.

## Risks / Trade-offs

- [Interactor tests cannot inject an exact current time] → Assert timestamps fall within normalized before/after bounds, while domain and repository tests use fixed timestamps for exact behavior.
- [Repository trait signature changes affect mocks and import call sites] → Update compile-time callers and add focused lookup-or-create persistence coverage.
- [Restore now returns a different `updated_at` than the selected historical snapshot] → Specify and test this behavior while preserving `created_at`; GraphQL shape remains compatible.

## Migration Plan

No database migration or data rewrite is required. Deploy the application changes together. Rollback restores the previous application behavior without modifying stored rows or schema.
